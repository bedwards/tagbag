When most people think about where their code lives, they think about GitHub. GitHub is the place where you push your commits, where you open pull requests for code review, where you file issues to track bugs and features, and where automated pipelines build and test your software every time something changes. It does all of these things under one roof, and that is both its greatest strength and its most significant constraint. You are renting space in someone else's building, and you cannot rearrange the walls.

TagBag exists because we wanted to own the building. Not in the abstract sense of "we value open source," but in the concrete sense of having the actual source code for every service running on our machine, being able to read any line of it, change any behavior, and submit those changes back upstream if they are good enough. The project is a composition of three independent open-source tools that together replicate the core experience of GitHub while giving us total control over every layer.

The first thing to understand is what GitHub actually provides, broken into its component parts. There is the git repository hosting itself, which is the most fundamental piece. Git is a distributed version control system, meaning every developer already has a full copy of the history on their own machine, but you still need a central place where everyone agrees the canonical version lives. GitHub adds a web interface on top of that central repository, letting you browse code, view commit history, compare branches, and read files without needing to clone anything locally. This is what people mean when they talk about the Code tab and the Commits page.

On top of that repository hosting, GitHub layers a code review workflow called pull requests. A pull request is not a git concept at all. Git knows about branches and merges, but the idea of requesting that someone review your changes before they get merged into the main branch is a social convention that GitHub formalized into a product feature. Pull requests give you a place to have line-by-line discussions about code, to request specific reviewers, to require that certain checks pass before merging, and to choose how the merge happens, whether as a regular merge commit, a squash, or a rebase.

Then there is issue tracking, which is conceptually separate from code hosting but tightly integrated with it in GitHub's design. An issue is a unit of work: a bug to fix, a feature to build, a question to answer. GitHub's issue tracker is deliberately simple, which is a design choice that works well for small projects but starts to feel limiting when you need sprint planning, kanban boards, priority fields, custom workflows, or any of the things that tools like Jira or Linear provide. The power of GitHub's approach is that issues live right next to the code, so you can reference issue number one hundred and twenty-three in a commit message and GitHub will automatically link them together.

Finally, there is continuous integration and continuous delivery, which GitHub calls Actions. When you push code, a pipeline of automated steps runs: compiling, testing, linting, deploying. Actions uses workflow files written in a markup language called YAML that describe what should happen and when. The key insight of Actions is that these workflows run in containers, which are lightweight isolated environments, so each step gets a clean slate.

TagBag replaces each of these pieces with a dedicated open-source project. For repository hosting and pull requests, we use Gitea, which is a single Go binary that provides an experience remarkably close to GitHub's. Gitea handles git hosting, pull requests with inline code review, branch protection rules, webhooks, a package registry, and even its own version of Actions. It stores its data in PostgreSQL, which is a relational database known for its reliability and SQL standards compliance, and it exposes a comprehensive application programming interface, or API, that covers every operation you can perform in the web interface. There is also a command-line tool called tea that lets you create pull requests, manage issues, and interact with Gitea without opening a browser.

For issue tracking and project management, we use Plane, which is a much more capable tool than GitHub's built-in issues. Plane is closer to Linear or Jira in its feature set. It gives you kanban boards, sprint cycles, modules for grouping related work, custom workflow states, priority levels, and a triage inbox for incoming issues. Plane is built with Django on the backend, which is a Python web framework, and React on the frontend. It also uses PostgreSQL for its database, and it exposes an API that lets you create and manage what it calls work items, which is its term for issues. Plane does not have an official command-line tool, so TagBag includes one that wraps its API.

For continuous integration, we use Woodpecker, which is a lightweight pipeline runner written in Go. Woodpecker has a server that coordinates work and an agent that actually executes pipeline steps inside containers. It integrates directly with Gitea, meaning when you push code to a Gitea repository, Woodpecker automatically picks up the pipeline definition from your repository and runs it. It uses its own YAML syntax for pipeline definitions, which is simpler than GitHub Actions but covers the same core concepts: steps, services, secrets, matrix builds, and conditional execution. Like the other two, it stores its data in PostgreSQL and has both an API and a command-line tool.

The architectural decision that ties all three together is that they share a single PostgreSQL database server. Not the same database, but the same server running four separate databases: one for Gitea, one for Plane, one for Woodpecker, and one for TagBag itself. The first three are straightforward — each service stores its own data in its own schema with no cross-contamination. The fourth database, the one that belongs to TagBag, is where the coordination layer lives. This is the place for the data that none of the individual tools would know about on their own: the mappings between a Plane work item and a Gitea pull request, the record of which webhook events have been processed, the unified activity stream that stitches together commits, issue updates, and pipeline results into a single timeline. It is not a duplication of what the other databases hold, but rather the connective tissue that makes three tools feel like one product. PostgreSQL was a hard requirement from the start because it is the most capable open-source relational database, and choosing projects that all support it as a backend means we can use a single, well-understood technology for all persistent state.

Everything runs through Docker Compose, which is a tool for defining and running multi-container applications. Our Compose file describes sixteen services: Gitea, all of Plane's components, which include the API server, background workers, a scheduler, a migration runner, a real-time collaboration server, a frontend web server, an admin panel, a public sharing interface, a reverse proxy, a Redis cache, a message queue, and object storage, plus the Woodpecker server and agent, and the shared PostgreSQL instance. The command to bring the entire stack up or tear it down is a single line.

What makes TagBag different from simply running these three tools side by side is the integration layer. Each project is a fork of the upstream repository, not a clone. Forking means we have our own copy on GitHub that we fully control, and we can push changes to it, create branches, and submit pull requests back to the original project. These forks are then included in our main repository as git submodules, which is a mechanism that lets one repository embed a specific commit from another repository. This gives us reproducible builds — anyone who clones our repository and initializes the submodules gets exactly the same version of every component.

Authentication is unified through Gitea acting as an identity provider using the OAuth 2.0 protocol. OAuth is a standard that lets one application delegate authentication to another. In practice, this means you create an account in Gitea, and when you visit Plane or Woodpecker, they redirect you to Gitea's login page. Once you are logged in there, Gitea sends you back to whichever service you were trying to reach, along with a token that proves who you are. If you already have an active Gitea session, these redirects happen instantly without any password prompt, which creates the feeling of a single sign-on experience without needing any additional authentication infrastructure. Plane's codebase already includes a built-in Gitea OAuth provider, and Woodpecker was designed from the ground up to authenticate through a git forge like Gitea.

The command-line interface brings all three services under one tool. Rather than remembering three different commands with three different configuration mechanisms, you run tagbag followed by the service you want to interact with. Tagbag plane lets you create and manage work items, cycles, and modules. Tagbag gitea lets you create repositories, open pull requests, and manage branches. Tagbag ci lets you trigger pipelines, view logs, and manage secrets. There is also tagbag login, which walks you through setting up tokens for all three services and stores them in a single credentials file, and tagbag whoami, which shows your identity across all three services at once. For operations that the unified tool does not cover, you can pass through to the native tools directly — tagbag gitea tea and tagbag ci wp hand off to the Gitea and Woodpecker command-line tools with their full feature sets.

The deeper value of this arrangement becomes clear when you think about what it means to hack on the tools themselves. Because every service is built from source rather than consumed as a prebuilt container image, you can open any file in any of the three codebases, understand how something works, and change it. Gitea is a Go application with a clean separation between its routing layer, its business logic, and its database models. Plane is a Python and TypeScript monorepo where the backend follows Django's conventions and the frontend uses React with MobX for state management. Woodpecker is another Go application with an interesting split between its server, which handles coordination, and its agent, which handles execution, communicating over a protocol called gRPC, which is a high-performance remote procedure call framework developed by Google. Each project has well-documented extension points: adding a new API endpoint, creating a new webhook event, building a pipeline plugin, or integrating a new git hosting provider. Having the source means you can wire these systems together at the deepest level, not just through their public APIs but by modifying how they work internally.

And that wiring is the real point. The goal is not to run three tools side by side and call it a day. The goal is to create a unified experience that feels as seamless as GitHub itself, but one that you own completely. That means a single sign-on where you authenticate once and move freely between code review, issue tracking, and pipeline management without ever seeing a second login screen. It means a command-line interface where one tool, one set of credentials, and one mental model covers everything: filing an issue, opening a pull request, checking why a build failed, all without switching contexts or remembering which tool handles which concern. It means webhooks that automatically link a pull request to the issue it closes, that update a work item's status when a branch is created for it, that post deployment results back to the issue tracker so the full lifecycle of a change is visible in one place.

On the web side, the vision is equally ambitious. A shared navigation layer that spans all three services gives you the feeling of a single application even though the underlying engines are distinct. Consistent branding, shared color palettes, and a common header mean that moving from your project board to your repository to your build dashboard feels like navigating within one product, not jumping between three. The integration goes deeper than cosmetics: cross-service search, unified activity feeds, and a landing dashboard that aggregates the state of your issues, pull requests, and pipelines into a single view. This is what it means to have a single pane of glass, a place where you can see everything that matters about your project at once without opening three tabs and mentally stitching the picture together.

What makes this possible is the fork-and-own model. You are not limited to what each project's maintainers have chosen to expose through configuration or plugins. If Gitea's template system needs a shared navigation bar, you add one to the templates in your fork. If Plane needs to display pipeline status on an issue, you add an API call to the frontend in your fork. If Woodpecker needs to post richer context back to Plane when a deployment finishes, you modify the forge integration in your fork. These are changes you can make because you have the source, because you build from source, and because your forks are submodules that you control. And when your changes are good enough, you submit them upstream, improving the tools for everyone.

The deeper trade-off here is not complexity in exchange for control, because the whole point of the unified layer is to eliminate the complexity that the user sees. The trade-off is engineering effort upfront in exchange for a development environment that works exactly the way you want it to, that cannot be taken away from you by a pricing change or a terms-of-service update, and that improves every time you invest in it. You are not renting space in someone else's building. You are building your own, and every wall is where you want it.
